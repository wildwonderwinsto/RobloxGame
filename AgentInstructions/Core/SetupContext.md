# Roblox Project Setup & Tooling Context

**Purpose:** reference context for any AI agent or contributor working in this repo. Describes the toolchain, folder layout, file-naming conventions, and workflows this project uses. Pairs with `Structure.md` (code architecture rules).

**Source reference:** based on the professional workflow walkthrough in this video: https://www.youtube.com/watch?v=IJDg6tRJmHo

---

## 1. Why This Setup Exists

Roblox Studio's built-in editor works, but it has no real version control, no rich IntelliSense, no fast refactoring, and no linting. Studio alone doesn't scale past a hobby project or a solo dev. This setup replaces "script directly in Studio" with an external, version-controlled, tool-assisted workflow while still live-syncing into Studio for testing.

---

## 2. Toolchain Overview

| Tool | Role |
|---|---|
| **Roblox Studio** | Engine/runtime — where the game is actually built, tested, and played |
| **VS Code** | Primary code editor — where all `.luau` source is written |
| **Rojo** | Live-syncs the VS Code file tree into Roblox Studio instances, and vice versa |
| **Rokit** | Toolchain manager — installs and pins versions of Rojo, Wally, and other CLI tools |
| **Wally** | Package manager — installs and links reusable third-party/shared Luau libraries |
| **Git + GitHub** | Version control and remote backup/collaboration |
| **Selene** | Linter — flags bad practices/mistakes the compiler won't catch |
| **StyLua** | Auto-formatter — enforces the official Luau style guide on save |

**Agent note:** never edit compiled/runtime state directly in Studio as the source of truth. Studio is a *sync target*. The VS Code file tree under `src/` is the source of truth.

## 3. Repo / Folder Layout

```
Roblox Projects/              -- top-level folder holding all projects (e.g. under Documents)
└── <game-name>/               -- this repo root
    ├── default.project.json   -- Rojo project map (source of truth for Studio hierarchy)
    ├── wally.toml              -- package manager dependency manifest
    ├── wally.lock               -- generated, do not hand-edit
    ├── rokit.toml               -- pinned CLI tool versions
    ├── selene.toml              -- linter config
    ├── .gitignore
    ├── Packages/                 -- generated by `wally install`, client/shared packages
    ├── ServerPackages/            -- generated by `wally install`, server-only packages
    └── src/
        └── ...                    -- see Structure.md for internal structure
```

## 4. File Naming Convention (Rojo)

Rojo infers script type from the filename suffix:

| Suffix | Compiles to |
|---|---|
| `Name.server.luau` | `Script` (server) |
| `Name.client.luau` | `LocalScript` (client) |
| `Name.luau` | `ModuleScript` |

**Agent rule:** always name new files according to this convention. Never create a bare `.lua` file — this project uses `.luau`.

---

## 5. `default.project.json`

- Defines how the `src/` file tree maps onto the Roblox instance hierarchy (`ReplicatedStorage`, `ServerScriptService`, `StarterPlayer`, etc.).
- Any folder Wally generates (`Packages/`, `ServerPackages/`) must be registered here — typically `Packages` under `ReplicatedStorage` and `ServerPackages` under `ServerScriptService` — or Rojo/Studio won't see them.
- Remove default instance entries under `workspace` unless this project intentionally wants Rojo to manage baseplate/world instances.

---

## 6. Package Management (Wally)

- Dependencies are declared in `wally.toml` under `[dependencies]` (client/shared) or `[server-dependencies]` (server-only, e.g. a data-persistence library).
- Preferred key casing for dependency names: **PascalCase**.
- Commands:
  - `wally install` — installs/updates packages into `Packages/` and `ServerPackages/`.
  - `wally package-types` — exports Luau types for installed packages (Wally does not do this automatically; run it after every install/update or IntelliSense for packages will be missing).
- After adding a new package, re-run `wally install` and `wally package-types`, and confirm the package folders are visible in Studio via Rojo sync.

---

## 7. Rokit (Toolchain Manager)

- Manages pinned versions of Rojo, Wally, and other CLI tools for reproducible environments across contributors.
- Setup commands:
  - `rokit init`
  - `rokit add rojo`
  - `rokit add wally`
  - `rokit add wally-package-types`
- Run `rokit init && rojo init` when bootstrapping a brand-new project from scratch.

## 8. VS Code Configuration

Required extensions:
- **Luau Language Server** — Luau language support.
- **Rojo** — connects VS Code to Studio; provides the sync UI (`Ctrl+Shift+P` → "Rojo").
- **Selene** — linting.
- **StyLua** — formatting.

Recommended settings:
- Format-on-save enabled, using StyLua as the formatter.
- Luau Language Server → enable **auto-imports**, so typing a package/service name auto-suggests and writes the full `require(...)` path or `game:GetService(...)` call.

---

## 9. Live Sync Workflow

1. In VS Code, open the Rojo panel and start the server (green arrow / "Start Server").
2. In Roblox Studio, open the Rojo plugin and click **Connect**.
3. Accept any incoming changes when prompted.
4. From then on, saving a file in VS Code live-syncs it into Studio automatically.

**Agent rule:** treat Studio's current session state as disposable. If Studio and the VS Code tree disagree, the VS Code tree (and `default.project.json`) wins — reconnect Rojo and accept the sync rather than hand-editing Studio.

---

## 10. Linting & Formatting

- **Selene** config lives in `selene.toml` at repo root (minimal — points at the Roblox standard library definitions). Selene catches logic/style issues the compiler won't (e.g. shadowed variables, unused locals).
- **StyLua** formats every file on save according to the official Luau style guide. Don't hand-format against StyLua's output — let it own formatting.

---

## 11. Git & GitHub Workflow

- `.gitignore` must exclude generated/local artifacts, at minimum:
  - `Packages/`
  - `ServerPackages/`
  - `wally.lock`
  - `sourcemap.json`
- Standard flow:
  1. Make changes.
  2. Stage + commit locally with a short, meaningful message once a unit of work is done (via VS Code's Source Control tab or CLI).
  3. Push/publish to GitHub to back up and share.
- **Branching:** create a feature branch per feature/fix (e.g. `feature/tank-class`). Changes on a branch don't affect other branches until merged; merge via a pull request on GitHub, then pull the merged changes back into the local repo.
- Only sign in to Git once per machine; use the same email as the linked GitHub account.

## 12. Agent Directives Checklist

When working in this repo:

- [ ] New scripts use `.server.luau` / `.client.luau` / `.luau` suffixes correctly — never `.lua`.
- [ ] Any new top-level folder that should sync to Studio is registered in `default.project.json`.
- [ ] New third-party dependencies are added to `wally.toml` (correct dependency section for client vs. server), followed by `wally install` and `wally package-types`.
- [ ] Never hand-edit `wally.lock`, `Packages/`, `ServerPackages/`, or `sourcemap.json` — these are generated.
- [ ] Formatting is left to StyLua; don't fight its output style.
- [ ] Commits are scoped and messaged per logical change; new features branch off rather than committing directly where inappropriate.
- [ ] Studio is never treated as the source of truth — `src/` + `default.project.json` are.
